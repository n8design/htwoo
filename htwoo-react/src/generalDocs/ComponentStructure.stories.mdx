# Component Structure Overview

This documentation is meant to help you understnad how the components are designed so that you can more easily understnad how to extend them.
The image below shows a `schematic` (if you will) for a general component called "MyTest".

Below the image is descriptions of the various aspects of the component.

1. We have a common set of standard properties that all the HTWOO components extend. We are importing that interface here.
1. The property interface defined for the component should extend the common property interface which includes a property called `rootElementAttributes` as well as a string property `dataComponent`. A major theme to this library is that it should utilize standard HTML/CSS elements and attributes and that the developer should be able to easily extend those elements and attributes. To that end every component at least provides a property that can augment the attributes of the root HTML element of the component. By default it's generically typed, but each component should adjust the typing based on the actual root element for the component so we include it in the property defintion so we can be reminded to change the typing. All other component properties are added to the properties interface and as much as makes sense `enums` are defined for any properties that have a fixed set of appropriate values.
1. Most components don't require state, but in the case where it does a state inteface and factory interface have been defined.
1. By default all components in the library are PureComponents. PureComponents don't implement the `shouldComponentUpdate` method but instead handles a shallow compare for you. By doing so we reduce the amount of code to implement and improve performance. There are exceptions to when this type of component should be used (namly when properties or state would require a deep compare) and you can read more about it in the [ReactJS documentation](https://reactjs.org/docs/react-api.html#reactpurecomponent). If you are contributing a component and believe that you should implement your own instance of `shouldComponentUpdate` please feel free to change to a normal component.
1. We believe strongly in logging, each component defines a LOG_SOURCE string that is utilized in all try/catch blocks to help your debug your components and know where the error is coming from. By utilizing the fancy little ðŸ’¦ icon we also help filter the browser console log so it's easy to spot where the HTWOO component has logged a message.
1. Sticking with our theme of configurability, we do however allow you to provide your own LOG_SOURCE value to make your messages cohesive by overriding the value using the `dataComponent` property.
1. Via the `rootElementAttributes` you can add additional classes to the component. If you do so your classes will be appended to the class list *AFTER* the HTWOO classes. This is key for overriding and adding additoinal accissibility properties to the element.
1. We add a data-component attribute to the root element of every component. Like with the browser console log this let's you spot your components in the DOM when inspecting the page. Also we spread any of the attributues the user has added via the `rootElementAttributes` property, adding the combined classes after so that it will override the inejcted class.
1. We've added try/catch blocks in all render functions to keep the code executing, and this is where console logging comes into play.
